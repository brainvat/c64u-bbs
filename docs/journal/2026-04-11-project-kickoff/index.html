The Vision

Today we kicked off c64u-bbs — a project with one ambitious goal: anyone who owns a Commodore 64 Ultimate should be able to clone this repository, run a setup script, and have a fully working C64-hosted BBS running on their machine in minutes. No SD card shuffling. No manual file transfers. Everything happens over the network.

The golden path targets the C64 Ultimate (the FPGA-based C64 reimplementation by Gideon's Logic) and communicates with it entirely through its REST API on port 80 and FTP on port 21. Secondary goals include supporting the same BBS on a VICE64 emulator and even on original C64c hardware with physical drives and REU.

What We Built Today

In a single session, we went from an empty repo to a working Python CLI that talks to a real C64 Ultimate:

  • Repository scaffolding — .gitignore, git hooks for secrets scanning (pre-commit) and docs consistency (pre-push), automated docs/TOC system, journal system
  • Security guardrails — 22 regex patterns scanning every commit and push for passwords, API keys, tokens, private keys, and connection strings. This is a public repo — no mistakes allowed
  • Python package (c64u-bbs v0.1.0) — installable via pip with a c64u CLI entry point
  • C64U REST API client — httpx-based client with X-Password authentication, typed error handling, and a DeviceBackend Protocol for future VICE support
  • Config system — cascading config (CLI flags > env vars > ~/.config/c64u-bbs/config.toml)
  • install.sh — one-command setup that creates venv, installs deps, sets up hooks, and smoke tests the CLI
  • Test suite — 8 tests with mocked HTTP transport, all passing

First Contact with C64U-Bean-Thad

The highlight of the day — our first successful API call to a real C64 Ultimate on the local network:

$ c64u --host 192.168.1.172 info ┌───────────┬────────────────┐ │ Product │ C64 Ultimate │ │ Firmware │ 3.14 │ │ FPGA │ 121 │ │ Core │ 1.47 │ │ Hostname │ C64U-Bean-Thad │ │ Host │ 192.168.1.172 │ │ Unique ID │ 25FF7F │ └───────────┴────────────────┘

We discovered along the way that the actual API response field names differ slightly from the official documentation (firmware_version vs version). A good reminder that testing against real hardware matters — docs can lag behind firmware.

The Road Ahead

Our v1 plan has six phases, each building on the last:

  • Phase 1 — Foundation + device info ✓ DONE
  • Phase 2 — Drive management (mount/unmount D64/D71/D81) + PRG runner
  • Phase 3 — FTP file management (browse, upload, download)
  • Phase 4 — Device config + network discovery
  • Phase 5a — PETSCII auto-answer proof-of-concept (prove the modem pipeline)
  • Phase 5b — Full BBS deployment (deploy real BBS software, accept telnet connections)

The end goal: c64u bbs deploy and you have a working BBS accepting connections over TCP, routed through the C64U's ACIA modem emulation to classic Commodore 64 BBS software. The 1980s, served over modern networking.

Key Decisions

  • Python + httpx (sync) — CLI-first, no async complexity. Can migrate later if needed.
  • DeviceBackend Protocol — keeps the door open for VICE without over-engineering now
  • Auto-answer before real BBS — prove the modem pipeline with a tiny program first, then layer on real BBS software
  • docs/journal/ excluded from TOC — journals live inside docs/ for GitHub Pages but are managed separately from project documentation
