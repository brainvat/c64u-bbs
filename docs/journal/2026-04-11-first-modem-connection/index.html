The Milestone

Today we achieved the first successful modem connection through the C64 Ultimate. From a Mac running our c64u CLI tool, we configured the modem, uploaded a BASIC program, and connected to it over TCP — proving the entire BBS pipeline works end-to-end.

1: C64U-BBS AUTO-ANSWER 3: WAITING FOR CONNECTION... 4: CONNECTION DETECTED! 5: WELCOME SENT! 6: DISCONNECTED. 7: WAITING FOR NEXT CONNECTION...

The screen RAM dump above — read directly from the C64U via the REST API's DMA read endpoint — shows the complete auto-answer lifecycle: the program waits for a connection, detects the incoming TCP call, sends a PETSCII welcome screen through the ACIA, waits, disconnects, and loops back for the next caller.

How It Works: The ACIA Approach

The C64 Ultimate emulates a 6551 ACIA (Asynchronous Communications Interface Adapter) at the SwiftLink address $DE00. This is the same interface that real SwiftLink cartridges used in the 1980s and 90s for modem communication. The C64U maps TCP connections through this emulation — when someone connects to the listening port, the C64 sees it exactly as it would see an incoming phone call on a Hayes modem.

AddressRegisterOur Use
$DE00 (56832)Data RegisterWrite bytes to send to TCP client
$DE01 (56833)Status RegisterPoll bit 3 (TDRE) to detect connection
$DE02 (56834)Command Register$09: DTR active, no IRQ, RTS low
$DE03 (56835)Control Register$1F: 8N1, 19200 baud (SwiftLink)

The program writes individual bytes to the Data Register ($DE00) to transmit PETSCII characters to the connected terminal. Each byte goes through the ACIA to the C64U's modem emulation layer, which forwards it over the TCP socket to the connected client.

The Auto-Answer BASIC Program

Rather than hand-assembling 6502 machine code, we took a pragmatic approach: a BASIC program with direct ACIA register access via PEEK and POKE. This is tokenized into a valid C64 PRG file by our Python tokenizer (autoanswer.py) and uploaded to the C64U via the REST API.

Key lines from the source (autoanswer.bas):

; Initialize ACIA: command=$09, control=$1F 20 POKE 56834,9:POKE 56835,31 ; Poll status register bit 3 (TDRE) - wait for connection 30 S=PEEK(56833) AND 8:IF S=0 THEN 30 ; Send PETSCII data byte-by-byte through ACIA 210 READ B:IF B=-1 THEN 300 220 POKE 56832,B ; Hang up: reset ACIA command register 400 POKE 56834,0

The PETSCII welcome screen is stored in DATA statements as raw byte values. PETSCII control codes set colors (159=cyan, 5=white, 30=green, 158=yellow, 154=light blue) and 147 clears the screen. The full source with register documentation is saved in src/c64u_bbs/bbs/autoanswer.bas.

Building the BASIC Tokenizer

One of the more interesting challenges was writing a BASIC tokenizer in Python that produces valid C64 PRG files. The tokenizer converts BASIC source text into the binary format the C64 expects: a 2-byte load address ($0801), then lines consisting of a next-line pointer, line number, tokenized content, and a null terminator.

We hit three tokenizer bugs during testing, all caught by reading the C64U's screen RAM via DMA:

  • AND keyword — SAND8 wasn't splitting into S AND 8. Fix: add spaces in the BASIC source
  • = operator — the alpha-predecessor guard was blocking = from being tokenized after variable names like S. Fix: only apply the guard to alphabetic keyword tokens, not operators
  • DATA statements — DATA -1 was tokenizing the minus sign. Fix: track DATA context and emit literal bytes until colon or end of line

The ability to read screen RAM remotely via GET /v1/machine:readmem?address=0400&length=1000 was invaluable for debugging — we could see the exact error messages on the C64 screen without being in the same room as the hardware.

The Full Pipeline

The complete data path, proven working today:

Mac (c64u CLI) ↓ REST API: PUT /v1/configs/Modem%20Settings/Listening%20Port C64U Modem Config (port 6400, ACIA at $DE00, SwiftLink) ↓ REST API: POST /v1/runners:run_prg C64 runs BASIC program, initializes ACIA ↓ Poll ACIA status register TCP client connects to port 6400 ↓ C64U modem emulation triggers ACIA BASIC program detects connection, sends PETSCII via POKE 56832,B ↓ ACIA → modem emulation → TCP socket Terminal displays welcome screen

Choosing a BBS: What's Next

With the modem pipeline proven, the next step is deploying actual BBS software. This is Phase 5b of our plan, and it involves some important decisions:

  • Color 64 BBS — one of the most popular C64 BBS systems, widely available as D64 images. Good SwiftLink support. Strong candidate for first deployment.
  • CBBS — the original Ward Christensen-style BBS. Historically significant, simpler to configure, but more limited.
  • Image BBS — feature-rich, actively maintained by the retro community, supports multiple drive types. More complex to set up but the most capable option.
  • C*Base — another popular option with good PETSCII graphics support.

Key considerations for our choice:

  • SwiftLink compatibility — the BBS must work with the ACIA at $DE00, not just user-port modems
  • Redistribution rights — this is a public repo, so we need software that is freely distributable or downloadable from a known archive
  • Automation — we need to be able to configure the BBS non-interactively (or at least script the initial setup) so c64u bbs deploy can work end-to-end
  • Single-drive operation — many BBS systems expect two drives; we should start with one that works well on a single 1541

The goal remains: c64u bbs deploy color64 and you have a working BBS accepting telnet connections, all from a single command.
