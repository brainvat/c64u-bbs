Image BBS: The Long Road to One-Click Deploy

April 12, 2026
« Journal « Home

The Grand Mission

The goal is deceptively simple: type c64u bbs deploy imagebbs and have a working BBS accepting telnet connections on a real Commodore 64. One command. Behind that one command is a pipeline that resets the C64, uploads two 800KB disk images, configures dual 1581 drives, injects a BASIC boot loader, and waits for the BBS to come online.

Today we spent many, many hours building that pipeline. Most of it felt like nothing was happening. But every dead end, every wrong turn, every "hold the phone" moment taught us something about how a 40-year-old BBS interacts with modern emulation hardware.

The Dead Ends (A Brief History)

This is the third BBS we've tried. The first two were educational failures:

  • Color 64 BBS v7.37 — Research agents confidently told us it had SwiftLink support. It doesn't. User-port only, 2400 baud max. Lesson: always verify claims against actual software.
  • C*Base v3.3.7 — Has SwiftLink support, GPL licensed, everything looked perfect. But its configure program has a broken save function — it silently fails to write config to disk. We proved C64U disk writes work fine with a simple OPEN/PRINT#/CLOSE test. The product itself was borked.
  • Image BBS v3.0 — February 2026 release from the 8-Bit Boyz community. Actively maintained, SwiftLink built in, two-disk D81 setup. This is the one.

What Actually Happened Today

We built the complete deploy pipeline: download script, catalog entry, FTP upload, drive configuration, boot loader generation. Then we hit the C64U for real.

The first boot attempt failed with ?FILE NOT FOUND — the boot file is IMAGE 3.0 (uppercase), not image 3.0 as noted in our plan. We read the actual D81 directory with Python to find the correct name. Classic PETSCII casing issue.

Then came the setup wizard. Fifteen prompts for sysop info, six pairs of device/drive assignments, clock settings, RS-232 type selection. Every answer matters and none of it is documented well enough to automate yet.

The Plus-File Disaster

After the setup wizard, the BBS crashed: ?LOAD ERROR IN 3096, trying to chain-load I/SU.CONFIG. We chased this bug for a long time. Theories included:

  • The / character confusing 1581 subdirectory parsing
  • ACIA initialization interfering with disk I/O
  • Custom ML loader incompatibility with C64U drive emulation
  • Filename case sensitivity

The actual cause? The Plus-File device must be set to device 8 (same as the Program disk), not device 9. This is nowhere in the sysop guide. A quick search of the 8-Bit Boyz forum found the answer in 30 seconds.

Lesson learned: when you've tried two workarounds and you're about to start a third, STOP. Google it. Ask the community. Ask the human for an idea. Don't chase the wascally wabbit around and around the same tree.

The Triumph

After fixing the Plus-File assignment, Image BBS loaded completely. We configured the modem through the BBS's own IM Configuration Editor — brand: SWIFTLINK, author: CMD, RS-232 type: SwiftLink ($DE00), all the Hayes AT command settings.

And then:

Connecting to 192.168.50.124:3000... CONNECTED ATA IMAGE BBS 64 V3.0 (C) 2025 CONNECTION CONFIRMED: 19200 HIT YOUR BACKSPACE/DELETE KEY 1 TIME 1:

Image BBS answered an incoming call on real C64 hardware. The full pipeline works: Mac → REST API → C64U modem emulation → ACIA at $DE00 → Image BBS → PETSCII welcome screen → TCP socket → terminal.

The Unfinished Business

The deploy pipeline works mechanically, but two issues remain:

  • Deploy-time modem config hang — calling configure_modem() via the C64U API right before boot puts the ACIA in a state that causes Image BBS to hang during sub.modem initialization. Skipping the modem config (relying on flash-saved settings) works. Same D81 images run perfectly in VICE. Root cause TBD.
  • Login crash — remote callers can connect and enter their handle, but login processing crashes at sub.misc line 60044. Likely corrupted user records from hung sessions. Needs clean golden D81 images.

The New Rule

We added a rule to CLAUDE.md today:

Hold the Phone: When a workaround fails twice, stop. Do not attempt a third. State the problem simply, ask if the conclusion even makes sense, and ask the human for an idea. Never accept absurd conclusions without challenging them. Apply Occam's razor before inventing elaborate theories.

This rule exists because we repeatedly burned time chasing symptoms instead of root causes. The Plus-File fix took 30 seconds of Googling after an hour of theorizing. The C*Base config save failure was the product being broken, not our disk writes. The "read-only drive" was never read-only — we proved it with a simple file write test.

The pattern: when something seems impossibly wrong, the simplest explanation is usually right. And the fastest path to that explanation is often asking someone who's seen it before.
